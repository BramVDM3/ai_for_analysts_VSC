# Exercise Explanation

We have a current library application. It is basic and it works, but it misses many features and creates frustration for different users.

We already have raw outputs from interviews, complaint emails, workshops, strategy discussions, data files, and incomplete models. The next step is to start the analysis.

## What We Need To Do

Analyze and structure all raw material:

- identify needs per stakeholder
- find frustrations with the current app
- identify things stakeholders like and want to keep
- discover needs for future features
- make conflicts and tensions visible
- decide which tensions need to be resolved

We also need to understand exactly how the current application works.

## Models

Create as-is models for the current situation, for example:

- current process model
- current class/domain model
- current user flow
- current data/API view

Then create to-be models for the proposed future situation.

## Needs Analysis

Create a needs analysis based on the raw material.

The analysis should show:

- stakeholder needs
- evidence from the raw material
- conflicts between stakeholders
- priorities
- open questions

## Epics And Stories

From the needs analysis, create epics and stories.

Stories should include:

- the story card
- the conversation, simulated by us
- clear acceptance criteria
- API designs where needed
- Gherkin tests
- user acceptance tests

## Mockups

Create mockups for important proposed features.

These can be:

- wireframes
- ASCII wireframes
- full HTML pages
- simple visual prototypes

## Final Presentation

At the end of the day, present the analysis to the board.

Create a concise PowerPoint using storytelling to convince the board to accept the proposal.

The presentation should explain:

- what the current app does
- what problems users have
- what the most important needs are
- which tensions exist
- what solution direction is proposed
- which epics should be built first
